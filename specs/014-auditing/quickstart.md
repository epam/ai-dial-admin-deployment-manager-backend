# Quickstart: Auditing Implementation

## Implementation Order

The feature should be built bottom-up in this sequence, with each layer verifiable independently.

### Step 1: Dependencies and Configuration

1. Add `hibernate-envers` and `uuid-creator` to `build.gradle`
2. Add Envers properties to `application.yml`
3. Update `JpaConfiguration` — add `AuditEntityPackage` to `@EntityScan`, add `transactionDateTimeProvider` bean with WARN-level fallback logging
4. **Verify**: `./gradlew clean build` — project compiles, existing tests pass

### Step 2: Transaction Timestamp Infrastructure

1. Create `TransactionTimestampContext` component
2. Create `TransactionTimestampAspect` (AOP `@Before` on `@Transactional`) with `@Order(Ordered.LOWEST_PRECEDENCE)`
3. Wire `DateTimeProvider` bean in `JpaConfiguration`; add `@EnableTransactionManagement(order = Ordered.LOWEST_PRECEDENCE - 10, proxyTargetClass = true)` to ensure the aspect fires inside the transaction boundary
4. **Verify**: `./gradlew testFast` — existing tests pass (timestamp behavior transparent to existing code)

### Step 3: Flyway Migrations

1. Create `V1.55__CreateAuditTables.sql` in all three dialect directories (H2, POSTGRES, MS_SQL_SERVER)
2. Include: `revinfo`, all 14 `*_aud` tables, `deployment_topics_aud`, `audit_activity`
3. **Verify**: `./gradlew testFast` — Flyway applies migration, Hibernate validates schema matches entities

### Step 4: Core Audit Entities and Enums

1. Create `ActivityType` and `ActivityResourceType` enums
2. Create `AuditRevisionEntity` with `@RevisionEntity`
3. Create `AuditActivityEntity`
4. Create `AuditActivityMapper` (entity class → resource type mapping)
5. Add `@Audited` to all 13 audited entity classes (base classes cover subclasses)
6. **Verify**: `./gradlew testFast` — schema validates, entities are recognized

### Step 5: Revision Listener

1. Create `AuditRevisionListener` implementing `EntityTrackingRevisionListener` and `ApplicationContextAware`
2. Implement `newRevision()` — extract author/email via `SecurityClaimsExtractor`; if author is null, fall back to `"unknown"` (when `Authentication` present) or `"system"` (when no `Authentication` — system operations); set timestamp from `TransactionTimestampContext`
3. Implement `entityChanged()` — build activities, handle deduplication
4. **Verify**: Write a test that creates/updates/deletes a deployment and asserts `audit_activity` rows appear with correct data

### Step 6: Activity Query Layer

1. Create `AuditActivityJpaRepository` in `dao.jpa` (extends `JpaSpecificationExecutor`)
2. Create `AuditActivityService` with specification-based filtering and field whitelist validation
3. Create domain model `AuditActivity` (uses enum types for `activityType`/`resourceType`)
4. Create persistence mapper (`PersistenceAuditActivityMapper` — MapStruct)
5. **Verify**: Write a service-level test that filters activities by type and resource

### Step 7: REST API

1. Create `PageRequestDto` (with `@Min`/`@Max`/`@NotBlank` validation), `PageDto` (web DTOs)
2. Create `AuditActivityDto`
3. Create `AuditActivityDtoMapper` (MapStruct)
4. Create `AuditActivityController` with POST and GET endpoints
5. Add OpenAPI annotations
6. **Verify**: Full integration test — create deployment, query activities endpoint, assert correct response

### Step 8: Documentation and Schema

1. Update `docs/configuration.md` with new Envers properties
2. Run `./gradlew generateDbSchema` to regenerate schema docs
3. **Verify**: `./gradlew clean build` — full test suite passes across all database dialects

## Key Gotchas

- **Naming strategy**: Spring Boot's `CamelCaseToUnderscoresNamingStrategy` lowercases Envers table/column names. Migration SQL must use lowercase `_aud`, `rev`, `revtype`.
- **JOINED inheritance**: Base `_aud` table holds `revtype`; subclass `_aud` tables have `rev` + PK + subclass columns only.
- **Listener is not a Spring bean**: `AuditRevisionListener` is instantiated by Hibernate, not Spring. Use `ApplicationContextAware` to access Spring beans.
- **TransactionTimestampAspect ordering**: Must fire INSIDE the transaction boundary (after `TransactionInterceptor` starts the transaction) because `registerSynchronization()` requires an active transaction. Fixed via `@Order(Ordered.LOWEST_PRECEDENCE)` on the aspect and `@EnableTransactionManagement(order = Ordered.LOWEST_PRECEDENCE - 10, proxyTargetClass = true)` on `JpaConfiguration`. The `proxyTargetClass = true` is required because `@EnableTransactionManagement` defaults to JDK interface proxies, which breaks constructor injection of concrete service classes.
- **H2 migrations diverge**: H2 uses `AUTO_INCREMENT` for `revinfo.id`; Postgres uses `GENERATED BY DEFAULT AS IDENTITY`; MS SQL uses `IDENTITY`.
- **Existing data**: Envers does NOT retroactively audit existing entities. Only future changes are tracked. This is expected.
- **Test infrastructure**: The `FunctionalTestSuite` dump/restore pattern restores the DB after each test. Audit tables should be included in the restore (or explicitly cleaned).
