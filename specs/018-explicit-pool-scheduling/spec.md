# Feature Specification: Explicit Node Pool Scheduling

**Feature Branch**: `018-explicit-pool-scheduling`
**Created**: 2026-05-11
**Status**: Implemented
**Capability**: deployments
**Input**: User description: "Node pool selector API need modification. Remove min/max node count, cpu, memory. Instead of configuring node selector programmatically, administrator passes selector, affinity, and tolerations explicitly per pool. Both nodeSelector and nodeAffinity must be supported."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Configure Pools With Explicit Scheduling Primitives (Priority: P1)

A platform operator wants to define which Kubernetes nodes each node pool maps to without the deployment-manager inferring it from a convention. Instead of one cluster-wide label key plus a per-pool name, the operator declares — directly on each pool — the full set of scheduling primitives that Kubernetes will honour: `nodeSelector` (simple key=value map), `affinity` (full Kubernetes affinity object, including `nodeAffinity` with required/preferred terms and `matchExpressions`), and `tolerations` (list of standard taint tolerations).

**Why this priority**: This is the headline change. Without the new configuration shape, the rest of the feature has nothing to consume. The previous convention (`{labelKey: poolName}` derived selector, fixed per-pool capacity numbers) is being removed entirely — there is no half-state where both coexist, so this story must land first.

**Independent Test**: Set `NODE_POOLS` to a YAML document listing pools where each pool carries any combination of `nodeSelector`, `affinity`, and `tolerations`. Start the application. Verify (a) startup succeeds, (b) the parsed configuration is internally exposed in the same shape as the input (round-trip), (c) the same configuration expressed as single-line JSON (YAML's syntactic subset) is accepted equivalently, and (d) configurations that violate the schema (unknown top-level fields, malformed K8s structures, duplicate pool names, blank name) fail startup with a descriptive error.

**Acceptance Scenarios**:

1. **Given** `NODE_POOLS` declares a pool with only a `nodeSelector` (e.g. `{accelerator-type: nvidia-a100}`), **When** the application starts, **Then** that pool is loaded and its scheduling primitives consist solely of the `nodeSelector` (no affinity, no tolerations).
2. **Given** `NODE_POOLS` declares a pool with only `affinity.nodeAffinity.requiredDuringSchedulingIgnoredDuringExecution` carrying `matchExpressions` (operator `In`, multiple values), **When** the application starts, **Then** that pool is loaded with the affinity intact and no nodeSelector.
3. **Given** `NODE_POOLS` declares a pool that combines `nodeSelector`, `affinity`, **and** `tolerations`, **When** the application starts, **Then** all three sets coexist on the pool and are applied together at deploy time (Kubernetes AND-semantics between nodeSelector and nodeAffinity, append-semantics for tolerations).
4. **Given** `NODE_POOLS` declares two pools with the same `name`, **When** the application starts, **Then** startup fails with an error identifying the duplicate.
5. **Given** `NODE_POOLS` declares a pool whose `affinity` or `tolerations` block does not conform to the Kubernetes schema (e.g. unknown operator, missing required field), **When** the application starts, **Then** startup fails with an error identifying the offending pool and field.
6. **Given** the previous configuration format (`maxNodes`, `cpu`, `memory`, `gpu` on a pool entry) is supplied, **When** the application starts, **Then** the unsupported fields cause a startup error — there is no silent fallback. A stray top-level `NODE_POOL_LABEL_KEY` environment variable is **ignored** (not rejected) — it is no longer consumed anywhere and its presence has no effect on scheduling.
7. **Given** `NODE_POOLS` is supplied as single-line JSON (the syntactic subset of YAML), **When** the application starts, **Then** the configuration is loaded equivalently to its multi-line YAML form — operators may freely choose either form.
8. **Given** `NODE_POOLS` is supplied as multi-line YAML with `#` comments interleaved between pools or fields, **When** the application starts, **Then** the comments are ignored by the parser and have no effect on the loaded configuration.
9. **Given** a pool's `id` is unchanged but its `name` is rewritten between two restarts (operator wants to rename "GPU Pool A" → "GPU A100 Pool"), **When** the application restarts and an existing deployment that references the pool by id is fetched or redeployed, **Then** the deployment still resolves to that pool's primitives and the response `nodePoolName` reflects the new label — renaming `name` is non-breaking by design.
10. **Given** two pool entries declare the same `id` (e.g. typo when adding a new pool), **When** the application starts, **Then** startup fails with an error identifying the duplicate id. Two pools sharing the same `name` (with different ids) also fail startup with a duplicate-name error — both fields are validated for uniqueness at startup.

### User Story 2 — Deployments Use The Pool's Explicit Primitives At Deploy Time (Priority: P1)

When a deployment that references a pool is activated, the deployment-manager projects the pool's `nodeSelector`, `affinity`, and `tolerations` verbatim onto the workload's pod template (or the equivalent slot on the CRD spec for NIM / KServe-Inference deployments). The previously derived `{labelKey: poolName}` node affinity is no longer synthesised — what ends up on the workload is exactly what the operator wrote in `NODE_POOLS`.

**Why this priority**: Co-priority with Story 1. Configuration without enforcement does nothing for the user; enforcement without the new config shape has nothing to apply. The two land together.

**Independent Test**: Pick a deployment that references a pool whose configuration includes (a) a `nodeSelector` with two keys, (b) an `affinity.nodeAffinity` with one required `matchExpressions` block (operator `In`, two values), and (c) two `tolerations`. Deploy it. Inspect the resulting Kubernetes workload (Deployment / Knative Service / NIMService / KServe InferenceService as appropriate for the deployment type) and verify the three sections appear on the pod template (or the corresponding CRD slot) byte-for-byte equivalent to the pool's configuration.

**Acceptance Scenarios**:

1. **Given** a deployment references a pool that defines only `nodeSelector`, **When** the deployment is activated, **Then** the workload's pod template carries that `nodeSelector` and no synthesised node affinity.
2. **Given** a deployment references a pool that defines only `affinity`, **When** the deployment is activated, **Then** the workload's pod template carries that affinity verbatim and no `nodeSelector`.
3. **Given** a deployment references a pool that defines all three (`nodeSelector`, `affinity`, `tolerations`), **When** the deployment is activated, **Then** all three are present on the workload — Kubernetes will AND the selector and affinity at scheduling time and apply tolerations independently.
4. **Given** a deployment previously activated with the old derived `{labelKey: poolName}` affinity, **When** the deployment is redeployed under this feature, **Then** the old derived affinity is replaced by the pool's explicit primitives (no leftover constraints from the previous scheme).
5. **Given** a deployment's stored `nodePool` is null (the user explicitly selected "Any", the create-time cascade resolved to null because no fallback default applied, or `NODE_POOLS` is empty), **When** the workload is activated, **Then** the deployment-manager applies no pool-derived scheduling primitives — the workload retains only whatever defaults the deployment type already supplies. Deploy-time logic reads the stored value verbatim; it does **not** consult fallback defaults (those apply only at create time — see Story 4). Null is a normal, fully-supported state regardless of how `NODE_POOLS` is configured.

### User Story 3 — Listing API Exposes The Pool's Scheduling Primitives (Priority: P2)

The `GET /api/v1/node-pools` endpoint returns each configured pool's `name`, `description`, and the scheduling primitives the operator declared (`nodeSelector`, `affinity`, `tolerations`). Capacity-like fields (`maxNodes`, `cpu`, `memory`, `gpu`) that the previous shape returned are removed from this response — live utilisation is now the domain of the separate `?includeUtilization=true` flag delivered by Feature 020.

**Why this priority**: Visible-surface change but not gating the deploy-path correctness; the admin UI can be updated independently once the underlying schema is stable. Lower than P1 because Stories 1 and 2 deliver the user value (correct scheduling); Story 3 makes the configuration auditable from the API.

**Independent Test**: Configure two pools — one with only a `nodeSelector`, one with a full `affinity` + `tolerations` block. Call `GET /api/v1/node-pools` (no query parameters). Verify (a) the response is an array of two entries, (b) each entry carries `name`, `description`, and only the scheduling primitives that pool declared (absent fields are omitted from the JSON, not returned as nulls or empty objects), (c) no `maxNodes` / `cpu` / `memory` / `gpu` fields appear, and (d) the response remains backwards-compatible with Feature 020's `?includeUtilization=true` query parameter — a `utilization` block is added when that flag is set, the static shape from this story is unchanged.

**Acceptance Scenarios**:

1. **Given** two pools are configured with different primitive subsets, **When** `GET /api/v1/node-pools` is called, **Then** each entry in the response reflects only the primitives that pool declared (no synthesised defaults, no empty `{}` placeholders).
2. **Given** any configured pool, **When** the endpoint is called, **Then** the response MUST NOT include `maxNodes`, `cpu`, `memory`, or `gpu` fields on the configured-pool shape.
3. **Given** the endpoint is called with `?includeUtilization=true`, **When** the response is returned, **Then** each pool entry carries the static shape from this story plus the `utilization` block defined by Feature 020 (this story does not change Feature 020's contract).

### User Story 4 — Stamp Fallback Defaults Onto New Deployments At Create Time (Priority: P2)

Most deployments will not carry an explicit `nodePool` selection — UI users create them without thinking about scheduling, and yet the platform operator still wants a sensible landing spot. The operator declares an optional **catch-all default** via `NODE_POOL_DEFAULT` and, when model-serving workloads need different hardware, an optional **model override** via `NODE_POOL_DEFAULT_MODEL` (applied to NIM and KServe-Inference deployments in place of the catch-all). At create time, if a deployment's payload omits `nodePool`, the system runs a cascade — `NODE_POOL_DEFAULT_MODEL` (for model workloads, if set) → `NODE_POOL_DEFAULT` (if set) → null — and **stamps the resolved value onto the deployment record**. The create response returns the stamped value; subsequent GETs return it; deploy-time logic reads it verbatim. The user always sees on the FE exactly which pool a deployment is on. Admin changes to either env var apply only to *new* deployments created after the change — existing deployments keep whatever was stamped on them. Migrating existing deployments to a new default is an explicit user action (edit the deployment), never a silent background effect.

**Why this priority**: Story 4 layers on top of Stories 1 and 2 without altering deploy-time projection mechanics. It is a create-time concern about defaulting + transparency, not a deploy-time concern about resolution. Same priority as Story 3 — both are admin-experience improvements rather than deploy-path-correctness gates.

**Independent Test**: Configure `NODE_POOLS` with at least two pools, set `NODE_POOL_DEFAULT=cpu_pool` and `NODE_POOL_DEFAULT_MODEL=gpu_pool`. Create one NIM deployment and one MCP deployment, both with `nodePool` omitted from the create payload. Verify (a) the create response for the NIM deployment contains `nodePool: gpu_pool` and the MCP deployment contains `nodePool: cpu_pool` — the resolved defaults are visible immediately; (b) a subsequent `GET` on each deployment returns the same stamped value; (c) the workloads deploy with the corresponding pool's primitives. Then change `NODE_POOL_DEFAULT_MODEL=gpu_pool_v2`, restart, **redeploy** the existing NIM deployment without editing its `nodePool`: the deployment still shows `nodePool: gpu_pool` and the workload still uses `gpu_pool`'s primitives — the admin change is **not** retroactive. Finally, create a fresh NIM deployment without `nodePool`: it is stamped with `gpu_pool_v2`.

**Acceptance Scenarios**:

1. **Given** `NODE_POOL_DEFAULT_MODEL=gpu_pool` is configured and `gpu_pool` exists in `NODE_POOLS`, **When** a NIM deployment is created with `nodePool` omitted from the payload, **Then** the create response returns the deployment with `nodePool: gpu_pool` and the deployment record stores `gpu_pool`.
2. **Given** `NODE_POOL_DEFAULT=cpu_pool` is configured (and `NODE_POOL_DEFAULT_MODEL` is unset), **When** an MCP deployment is created with `nodePool` omitted, **Then** the create response returns the deployment with `nodePool: cpu_pool` and the record stores `cpu_pool`.
3. **Given** both defaults are configured, **When** a deployment is created with explicit `nodePool: other_pool` in the payload, **Then** the record stores `other_pool` — neither fallback overrides an explicit choice.
4. **Given** `NODE_POOLS` is non-empty and neither default is configured, **When** a deployment is created with `nodePool` omitted, **Then** the cascade resolves to null and the record is stamped with null (= "Any"). The create response returns `nodePool: null` and the FE renders it as "Any" — the user sees transparently that no pool was selected.
5. **Given** only `NODE_POOL_DEFAULT=cpu_pool` is set (catch-all, no model override), **When** a NIM deployment is created with `nodePool` omitted, **Then** the record is stamped with `cpu_pool` — the catch-all applies to every workload type the operator did not specifically override.
6. **Given** only `NODE_POOL_DEFAULT_MODEL=gpu_pool` is set (model override, no catch-all), **When** a NIM deployment is created with `nodePool` omitted, **Then** the record is stamped with `gpu_pool`. **And** when an MCP deployment is created with `nodePool` omitted, **Then** the record is stamped with null (= "Any") — the model override does not apply to non-model workloads, no catch-all is configured, and the cascade terminates at null.
7. **Given** `NODE_POOL_DEFAULT_MODEL=ghost_pool` (or `NODE_POOL_DEFAULT=ghost_pool`) where `ghost_pool` is not present in `NODE_POOLS`, **When** the application starts, **Then** startup fails with an error identifying the offending env var and the missing pool name.
8. **Given** an existing NIM deployment carries `nodePool: gpu_pool` (stamped earlier under `NODE_POOL_DEFAULT_MODEL=gpu_pool`), **When** the operator changes the env var to `gpu_pool_v2` and that deployment is later redeployed without any payload edit, **Then** the deployment still carries `nodePool: gpu_pool` and the workload uses `gpu_pool`'s primitives. The admin change is **not** retroactive.
9. **Given** the operator wants to migrate an existing deployment to a new pool, **When** they edit the deployment's `nodePool` field (via the standard update endpoint) to the new value, **Then** the record updates to the new value and the next redeploy applies that pool's primitives — migration is an explicit, auditable per-deployment action.
10. **Given** `NODE_POOLS` is non-empty, **When** a deployment is **updated** with explicit `nodePool: null` in the payload, **Then** the stored value is replaced with null verbatim — the user has selected "Any" (default K8s scheduling, no pool primitives) and the system honours that choice. (On **create**, the cascade always runs for a null payload value, so "Any" is not directly selectable at create time — see FR-013.)
11. **Given** a deployment with `nodePool: foo` is updated via PUT-style with `nodePool` omitted (or with `nodePool: null`), **When** the request is processed, **Then** the stored value is cleared to null. Update is PUT-style for `nodePool` per FR-014 — there is no "absent means preserve" semantic; callers PATCHing other fields MUST include the current pool value in the payload.
12. **Given** a UI/FE wants to show the user what pool would be applied by default before they submit, **When** it calls `GET /api/v1/node-pools`, **Then** the response lists the full set of selectable pools (the `pools` array, possibly empty). The defaults env vars are admin-only and are not part of this response (FR-017). The FE renders a pool picker containing the configured pools **and** an "Any" entry (mapping to null on the wire). Pre-selecting a default before the user submits is no longer driven by reading the env-var values off the listing endpoint — the FE either leaves the picker on "Any" until the user makes a choice, or submits a tentative create with `nodePoolId: null` and reads the resolved stamped value from the create response. Either way the user sees a concrete value once they submit, never a blank required field.
13. **Given** `NODE_POOLS` is empty (no pools configured), **When** a deployment is created with `nodePool` omitted or explicit null, **Then** the request succeeds and the record stores null — the feature is effectively off, current pre-feature behaviour preserved.

### Edge Cases

- **Old config in the wild.** If an existing environment still has `NODE_POOLS` in the old shape (with `maxNodes`/`cpu`/`memory`/`gpu`), startup fails with a descriptive error pointing at the deprecated field — there is no silent migration of the document. A stray `NODE_POOL_LABEL_KEY` environment variable is silently ignored: it is no longer consumed anywhere in the application, so its presence has no effect on scheduling. Operators are required to rewrite their `NODE_POOLS` document to the new shape, but cleaning up the legacy env var is optional cosmetic work. Pool configuration changes already require an application restart.
- **Empty primitives.** A pool declared with neither `nodeSelector`, nor `affinity`, nor `tolerations` is **legal** — it is a pool whose deployments simply receive no pool-derived scheduling constraints. The listing endpoint returns the pool with `name`/`description` only. Deploy-time logic applies nothing.
- **Workload already carries its own scheduling primitives.** The deployment-manager owns the pod-template scheduling slots for its managed workloads; it replaces (does not merge) any existing nodeSelector/affinity at deploy time and appends pool tolerations to whatever default tolerations the deployment type already injects (e.g. NIM operator defaults). Assumption documented below.
- **Pool removed after deployment was saved.** Unchanged from Feature 016: stored deployments keep their `nodePool` name; the listing endpoint will not show the removed pool; create/update still rejects unknown pool names; redeploy of a deployment whose pool no longer exists fails with a server error (configuration drift). The backend does **not** mark dangling deployments in retrieval responses, expose an audit query, or implement a deprecation lifecycle. Detecting and rendering the dangling state is a FE concern — the FE already fetches both the deployment (carrying its stored `nodePool`) and the current pool list (`GET /api/v1/node-pools`); a one-line client-side comparison gives the same information any backend-derived field could, with no duplication.
- **Pool renamed.** Stored deployments are not auto-updated — they continue to reference the old name and will fail to redeploy until the operator either renames the pool back or updates the deployment to point at a current pool. This matches Feature 016 semantics.
- **Affinity collision between `nodeSelector` and `affinity.nodeAffinity`.** Both can coexist on a single pool by design (the user explicitly asked for both). Kubernetes scheduler ANDs them. The deployment-manager performs no consistency check between the two — if the operator writes a `nodeSelector` that no node matching the `affinity` could satisfy, pods will simply stay Pending. This is the operator's responsibility, not the system's.
- **Default points at a pool that is later removed.** If a default env var (`NODE_POOL_DEFAULT` or `NODE_POOL_DEFAULT_MODEL`) is set and the operator removes the named pool from `NODE_POOLS` on the next restart, startup fails. The pool-removed and default-referencing-it states are detected and reported together. There is no silent fallback to "no default". Note: existing deployments stamped with the now-removed pool are a separate concern — their stored `nodePool` will fail name-validation on the next update or redeploy (matches Feature 016 "pool removed after deployment was saved" semantics).
- **Only catch-all set.** When `NODE_POOL_DEFAULT` is set but `NODE_POOL_DEFAULT_MODEL` is not, every deployment created without an explicit `nodePool` — including NIM and KServe-Inference — is stamped with the catch-all. To carve out model workloads onto different hardware, the operator sets the model override.
- **Only model override set.** When `NODE_POOL_DEFAULT_MODEL` is set but `NODE_POOL_DEFAULT` is not, model deployments created without an explicit `nodePool` are stamped with the override. Non-model deployments created without an explicit `nodePool` are stamped with null (= "Any") — the model override does not apply to them, no catch-all is configured, and the cascade terminates at null per FR-018.
- **Same pool used as both env vars.** Legal but redundant — the same outcome is achieved by setting only `NODE_POOL_DEFAULT`. The system does not warn about this; if the operator wants both values to drift independently later, the redundancy is a feature.
- **Admin changes a default env var.** The change applies only to deployments **created** after the change takes effect (after restart). Existing deployments retain whatever was stamped at *their* creation time. Migrating existing deployments to a new default is an explicit per-deployment update — never automatic. This is the headline transparency property: every deployment displays the pool it actually uses, and that value does not change behind the user's back.
- **User wants to opt out of pooling entirely.** Supported via update: the user creates the deployment (cascade fills the configured default), then updates it with `nodePool: null` to land on "Any". The record stores null; deploy-time logic applies no pool primitives. On create, null in the payload is not directly selectable — the cascade always runs (see FR-013). Operators who want to *prevent* the opt-out enforce it via UI policy or organisational practice; this API does not gate the option.
- **Cascade resolves to null (no defaults configured).** When `NODE_POOLS` is non-empty but neither `NODE_POOL_DEFAULT` nor `NODE_POOL_DEFAULT_MODEL` applies to a workload, an omitted `nodePool` in the create payload is stamped with null. The user sees `nodePool: null` ("Any") in the create response — no rejection, no friction, but also no opaque pool assignment.
- **`nodePool` field omitted from an update payload.** Update is PUT-style for `nodePool` (FR-014): omitting the field is equivalent to sending null and clears the stored value. The create-time cascade never runs on update — the payload's value (or its null absence) is authoritative. Callers PATCHing other fields MUST include the current pool value to avoid clearing it. (This is a deliberate simplification of the original spec's absent-means-preserve mechanic: the FE always re-binds the picker on edit, and the simpler contract has fewer footguns at the API boundary.)
- **`NODE_POOLS` becomes empty after previously being non-empty.** Existing deployments referencing now-undefined pools will fail to redeploy (Feature 016 semantics, unchanged). To recover, the operator either re-adds the pools or the user clears the field via update to null ("Any"). New deployments created in this state are stamped with null automatically.

## Clarifications

### Session 2026-05-12 (post-implementation review)

- Q: Should the listing endpoint `GET /api/v1/node-pools` surface the currently-configured fallback defaults (`NODE_POOL_DEFAULT`, `NODE_POOL_DEFAULT_MODEL`)? → A: No. The listing response envelope is just `{ pools: [...] }`. The two default env vars are admin-only configuration and are not exposed via any public read endpoint. A FE that needs to pre-select a default in the creation picker treats the create response's stamped `nodePoolId` as authoritative — submit with null, read the cascade's resolved value back. This supersedes the earlier "Listing API reshaped to `{ pools, defaults }`" PR design and the `defaultId` / `modelId` field naming. FR-017 and AC12 are restated above to match.
- Q: Should the legacy cluster-wide `NODE_POOL_LABEL_KEY` env var continue to cause a startup error if still set? → A: No. Nothing in the application reads `NODE_POOL_LABEL_KEY` after this feature ships, so leaving the variable set has no functional consequence. The earlier "fail loudly" FR-004 / SC-003 framing is replaced with "silently ignored" — operators who fail to clean up the legacy env var suffer no penalty.
- Q: Should the DB column on `deployment` be renamed `node_pool` → `node_pool_id` to mirror the Java field rename? → A: No. The Java field is renamed (`nodePool` → `nodePoolId`) and the JSON wire field is renamed (`nodePool` → `nodePoolId`), but the DB column stays `node_pool` (mapped via JPA `@Column(name = "node_pool")`). Renaming an already-merged Flyway migration in place would cause checksum drift on every environment that has already applied V1.58; the Java-side rename gives the same code-readability benefit without the data-layer hazard.

### Session 2026-05-12

- Q: The initial implementation tracked "was the `nodePool` field present in the JSON payload?" via an overridden Jackson setter on both the DTO and the service model, to distinguish absent-on-create (cascade) from explicit-null-on-create (Any), and absent-on-update (preserve) from explicit-null-on-update (clear). The doubled flag mechanism felt opaque. Simplify? → A: Yes. The DTO and the service model both carry only a nullable `nodePoolId` — no presence flag, no intent flag. Whether to apply the create-time cascade is decided by the **service entry point**: the public `DeploymentService.createDeployment(...)` delegates to a private overload that takes a boolean `applyNodePoolDefaultIfEmpty`; the public path passes `true`, and the duplicate flow calls the private overload directly with `false` so the source's value is preserved verbatim. Create with a null payload value (whether omitted or explicit null) always runs the cascade; selecting "Any" at create is no longer directly expressible — users create first, then update `nodePool` to null. Update becomes PUT-style for `nodePool`: the payload value is authoritative; absent equals null equals clear. Trade-off accepted: callers PATCHing other fields must always include the current pool, which the FE already does. This supersedes the earlier-in-session "explicit null on create stores null verbatim" position; FR-013 / FR-014 / AC10 / AC11 are restated above to match.
- Q: A pool name is shown on the UI **and** stored on every deployment that uses the pool. Renaming a pool in `NODE_POOLS` orphans every existing deployment that referenced the old name (redeploy fails). Should the pool's identity be split from its display label? → A: Yes. Each pool entry now carries two fields: `id` (immutable machine identifier, required, unique) and `name` (human-readable display label, required, unique-recommended but reserved for display only). Deployments persist the pool **id** (column `deployment.node_pool_id`, model field `nodePoolId`, wire field `nodePoolId`). Pool resolution at deploy time and create-time-cascade validation are both by id. Renaming a pool's `name` is now safe and has no effect on existing deployments — they pick up the new label on the next read because the DeploymentDto exposes `nodePoolName` as a read-time resolution of `nodePoolId` against the current configuration. Renaming a pool's `id` remains breaking (same blast radius as removing the pool). `NODE_POOL_DEFAULT` and `NODE_POOL_DEFAULT_MODEL` reference **ids**. The `defaults` block of `GET /api/v1/node-pools` exposes `defaultId` / `modelId`. Throughout the remainder of this spec, statements about "the deployment's `nodePool` value", "stamping a pool name onto the record", and similar phrasing refer to the pool **id** (the wire field is `nodePoolId`); the spec's older "name" framing is preserved for historical context but is subordinate to this clarification.

### Session 2026-05-11

- Q: When duplicating an existing deployment, what value should the new deployment's `nodePool` carry? → A: Copy the source's `nodePool` verbatim; validation per FR-013 applies, so duplicating a deployment whose source pool no longer exists fails until the user selects a current pool.
- Q: When importing a deployment whose stored `nodePool` doesn't exist in the target environment's `NODE_POOLS`, what should happen? → A: Reject the import; surface every offending deployment together with its missing pool name. The operator must reconcile (rename pools, edit the export, or pre-create matching pools in the target) before retrying.
- Q: Should configuration also be accepted via `application.yml`, or only via environment variables? → A: Both, with Spring Boot's standard precedence (env vars override `application.yml`). Canonical property keys: `app.node-pools.pools`, `app.node-pools.default`, `app.node-pools.default-model`.
- Q: Should `nodePool: null` be allowed when `NODE_POOLS` is non-empty? → A: Yes — null is a first-class user-selectable value meaning "Any" (no pool primitives, default K8s scheduling). The FE picker MUST render "Any" alongside configured pools. This reverses the earlier Position B decision: the pool list is now a catalogue of named scheduling presets rather than an authoritative whitelist. Users have a guaranteed escape; operators who need mandatory-pool semantics enforce it via UI policy, not via this API.
- Q: Should `nodePool` be included in the deployment **export** payload? → A: No — `nodePool` is environment-specific configuration (pool names, defaults, and scheduling primitives differ between dev/staging/prod), so it does not travel with the deployment across environments. Export omits `nodePool` entirely; import treats every incoming deployment as if its `nodePool` were absent, triggering the target environment's normal create-time cascade (which stamps with the target's defaults or null). This supersedes the earlier Q2 decision to validate-and-reject on import.
- Q: How are `nodePool` changes audited? → A: Defer to Feature 014's existing deployment-CRUD audit coverage. The create call is a single user action; whether `nodePool` arrived in the payload from the user or got filled by the cascade is an implementation detail invisible at the audit layer. The audit event records the user actor + the persisted field values (including `nodePool`'s final value) for create and update operations. There is no separate "system stamping" audit event, no synthetic actor, and no new audit work in this feature.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `NODE_POOLS` configuration MUST be supplied as a **YAML document** representing a list of pool entries. Because every JSON document is also a valid YAML 1.2 document, an operator may supply the value as either multi-line YAML or single-line JSON without functional difference — the parser MUST accept both forms.
- **FR-002**: `NODE_POOLS` configuration MUST accept, per pool, a `name` (required, non-blank, unique across the list), an optional `description`, and any combination of an optional `nodeSelector` (map of string → string), an optional `affinity` (full Kubernetes Affinity object — `nodeAffinity`, `podAffinity`, `podAntiAffinity` accepted), and an optional `tolerations` (list of standard Kubernetes Toleration objects).
- **FR-003**: `NODE_POOLS` configuration MUST be parsed in strict mode — any field at any level of the document that is not part of the declared schema (e.g. `maxNodes`, `cpu`, `memory`, `gpu` on a pool entry, or an unknown field inside a K8s sub-structure) MUST cause a startup error that names the offending pool and field path.
- **FR-004**: The cluster-wide `NODE_POOL_LABEL_KEY` environment variable MUST NOT be consumed anywhere in the application. The derived `{labelKey: poolName}` selector convention is fully replaced by per-pool `nodeSelector` / `affinity` / `tolerations`, so any value set on `NODE_POOL_LABEL_KEY` is ignored and has no effect on scheduling. The system does NOT reject the legacy variable at startup — operators who fail to clean it up suffer no functional consequence, because nothing reads it.
- **FR-005**: The system MUST validate every pool's `nodeSelector`, `affinity`, and `tolerations` against the Kubernetes API model at startup. Schema violations (invalid enum values like a bad toleration `operator` or `effect`, missing required sub-fields, type mismatches) MUST fail startup with an error that names the pool and the offending path.
- **FR-006**: At deploy time, the system MUST apply the selected pool's scheduling primitives **verbatim** to the workload's pod template (or the CRD's pod-template equivalent for NIM / KServe-Inference deployments): the pool's `nodeSelector` replaces any pool-derived nodeSelector on the workload, the pool's `affinity` replaces any pool-derived affinity on the workload, and the pool's `tolerations` are appended to whatever tolerations the deployment type already supplies by default.
- **FR-007**: At deploy time, when the selected pool declares neither `nodeSelector` nor `affinity` nor `tolerations`, the system MUST apply no pool-derived scheduling primitives — the workload's scheduling slots remain whatever the deployment type's defaults produce.
- **FR-008**: At deploy time, the system MUST read the deployment's stored `nodePool` verbatim. If it is set, that pool's primitives are projected per FR-006. If it is null, no pool-derived primitives are applied — the workload retains only whatever defaults the deployment type already supplies. Deploy-time logic MUST NOT consult `NODE_POOL_DEFAULT` or `NODE_POOL_DEFAULT_MODEL`; fallback defaults are a **create-time** concern only (see FR-018).
- **FR-009**: When a deployment's `nodePool` is changed via the update endpoint and the deployment is redeployed, the workload's pool-derived scheduling slots MUST reflect the new pool's primitives (replace for nodeSelector and affinity; for tolerations, remove the previous pool's appended entries and append the new one's — the operator-supplied defaults remain).
- **FR-010**: When a deployment's stored `nodePool` is null (whether from explicit user selection of "Any", from the cascade terminating at null on create, or from an update to null) and the deployment is redeployed, the workload's pool-derived scheduling slots MUST be removed (nodeSelector and affinity reset to the deployment-type defaults; previously appended pool tolerations removed). Deploy-time logic MUST NOT consult fallback defaults to repair the null — the stored value is canonical and null is a legitimate, fully-supported state.
- **FR-011**: `GET /api/v1/node-pools` (without `?includeUtilization=true`) MUST return, per pool, `name`, optional `description`, and any of `nodeSelector` / `affinity` / `tolerations` the pool declared. Fields that were not declared MUST be omitted from the response JSON (not returned as `null` or `{}`). `maxNodes`, `cpu`, `memory`, `gpu` MUST NOT appear in the response.
- **FR-012**: `GET /api/v1/node-pools?includeUtilization=true` MUST continue to return Feature 020's `utilization` block on every pool entry, layered on top of the static shape defined by FR-011. This feature does not change the utilisation contract.
- **FR-013**: The deployment create endpoint MUST accept an optional `nodePool` field (name only — never the full primitives). When the payload includes an explicit non-null name, the system MUST honour the supplied value verbatim. When the payload's `nodePool` is null — whether omitted or explicitly set to null — the system MUST resolve a value via the create-time cascade defined in FR-018 and stamp the resolved value (a pool name or null) onto the deployment record. The only validation is name resolution: a non-null `nodePool` value MUST reference a pool that exists in the current `NODE_POOLS`, otherwise the request MUST be rejected with a 400 error. The create response MUST return the deployment with its `nodePool` field reflecting the stored value, so the caller sees the resolved default (or null) without having to re-fetch. Selecting "Any" (null) at create time is not directly expressible — when defaults are configured the cascade always fills a null payload value; to land a deployment on null the user creates it and then updates `nodePool` to null (FR-014).
- **FR-014**: The deployment update endpoint MUST accept a `nodePool` field with PUT-style semantics: the payload's value is authoritative. An explicit non-null name MUST replace the stored value subject to name-resolution validation (unknown names rejected with 400). An explicit null MUST clear the stored value to null ("Any"). The update flow MUST NOT trigger the create-time cascade and MUST NOT consult fallback defaults — those are a create-time concern only. Deployment retrieval endpoints MUST return the stored `nodePool` name; this feature does not embed the pool's primitives into the deployment payload. **Caller contract**: clients PATCHing unrelated fields MUST include the current `nodePool` value in the payload — omitting the field is equivalent to sending null and will clear the stored pool. The FE's deployment-edit form satisfies this by re-binding every field including the pool picker.
- **FR-015**: The system MUST accept two **optional, independent** environment variables. `NODE_POOL_DEFAULT` is the catch-all fallback used at create time for any deployment whose payload omits `nodePool`. `NODE_POOL_DEFAULT_MODEL` is a narrower override that, when set, takes precedence over `NODE_POOL_DEFAULT` at create time for deployments classified as model workloads per FR-016. Both env vars are independently optional — any of the four combinations (neither, only catch-all, only model override, both) is valid. The same pool MAY be used as the value of both env vars (legal but redundant). **Each set value MUST be validated against `NODE_POOLS` at application startup**, as part of the same startup configuration-validation pass that validates `NODE_POOLS` itself (FR-003, FR-005). If either env var references a pool name that is not present in the current `NODE_POOLS`, the application MUST refuse to start and MUST emit a startup error that identifies the offending env var (`NODE_POOL_DEFAULT` or `NODE_POOL_DEFAULT_MODEL`) and the missing pool name. The validation is unconditional — it runs on every startup, against a fresh or populated database alike, and there is no flag to bypass it. (Note: when `NODE_POOLS` is empty, any non-empty default env var necessarily references a missing pool and therefore fails this check.)
- **FR-016**: The system MUST classify each deployment type as either a **model workload** ({NIM, KServe-Inference}) or otherwise. "Model workload" is the only named classification — there is no symmetric "non-model" named category. Model workloads are the only workloads to which `NODE_POOL_DEFAULT_MODEL` applies at create time; every other workload falls through to `NODE_POOL_DEFAULT`. This classification is system-defined and not configurable; new deployment types added in the future MUST be classified in code as part of their introduction.
- **FR-017**: `GET /api/v1/node-pools` MUST return a response envelope of the form `{ "pools": [...] }`. The `pools` array MUST always be present in the response (serialised as `[]` when no pools are configured) so FE consumers can rely on its presence without null-checking. The listing endpoint MUST NOT surface the values of the two fallback-default environment variables — `NODE_POOL_DEFAULT` and `NODE_POOL_DEFAULT_MODEL` are admin-only configuration and are not part of the public listing contract. A FE that needs to pre-select a default in the deployment-creation picker treats the create response's stamped `nodePoolId` as authoritative: the user submits with `nodePoolId: null`, and the create response carries the resolved pool id (or null) that the system stamped via the cascade in FR-018. There is no separate read-side surface for the defaults.
- **FR-018**: At create time, when the create payload omits the `nodePool` field, the system MUST resolve a value via the following cascade, stopping at the first match: (1) `NODE_POOL_DEFAULT_MODEL` if that env var is set **and** the deployment is classified as a model workload per FR-016; otherwise (2) `NODE_POOL_DEFAULT` if that env var is set; otherwise (3) null (= "Any"). The resolved value (pool name or null) MUST be persisted to the deployment record (FR-013) and returned in the create response. The cascade always terminates successfully — there is no rejection path. The cascade runs **exclusively at create time**; it MUST NOT run on update (FR-014) and MUST NOT run at deploy time (FR-008).
- **FR-019**: A change to either `NODE_POOL_DEFAULT` or `NODE_POOL_DEFAULT_MODEL` (taking effect on application restart) MUST NOT cause the system to mutate any existing deployment's stored `nodePool`. Existing deployments retain the value stamped at their creation time; only deployments created **after** the change takes effect see the new default. Migration of an existing deployment to a new pool MUST be an explicit per-deployment update via FR-014 — never automatic.
- **FR-020**: The deployment **duplicate** operation MUST copy the source deployment's `nodePool` verbatim onto the new deployment. The create-time cascade (FR-018) MUST NOT run on a duplicate — the source's value (pool name or null) is treated exactly as if the user had typed it into the create payload. The resulting deployment is still subject to FR-013's name-resolution check, so duplicating a deployment whose source pool no longer exists in `NODE_POOLS` MUST fail with a 400 error until the user replaces the field — either with a current pool name or with null ("Any") — and re-submits.
- **FR-021**: The deployment **export** payload MUST NOT include the `nodePool` field — pool assignment is environment-specific configuration and does not travel with the deployment. The deployment **import** flow MUST NOT consume any incoming `nodePool` field; if a `nodePool` field is present in the import payload (e.g. from a legacy export or hand-edited file), it MUST be ignored. Every imported deployment goes through the target environment's normal create-time cascade (FR-018) — the result is a record stamped with the target environment's appropriate default for the deployment's workload classification, or null if no default applies. There is no cross-environment pool validation, no reconciliation step, and no rejection path: import succeeds regardless of differences in pool naming or configuration between source and target environments.
- **FR-022**: All three configuration values MUST be bindable from **both** `application.yml` (under the property keys `app.node-pools.pools`, `app.node-pools.default`, `app.node-pools.default-model`) **and** their corresponding environment variables (`NODE_POOLS`, `NODE_POOL_DEFAULT`, `NODE_POOL_DEFAULT_MODEL`). Spring Boot's standard precedence applies — environment variables override `application.yml` values when both are present. The strict YAML parsing (FR-003), Kubernetes-model validation (FR-005), and default-references-existing-pool startup check (FR-015) MUST apply identically regardless of source. Operators MAY mix sources (e.g. define pools in `application.yml`, override a single default via env var) without altering validation behaviour.

### Key Entities

- **Node Pool (configured)** — A scheduling preset declared in `NODE_POOLS`. Holds `id` (required, **immutable**, unique — the machine identifier referenced by deployments), `name` (required, unique-recommended human-readable display label, **safe to rename**), `description` (optional), and any combination of three Kubernetes-native scheduling primitives: `nodeSelector` (key=value map), `affinity` (full Kubernetes Affinity object — `nodeAffinity` with required/preferred terms, optional `podAffinity` / `podAntiAffinity`), `tolerations` (list of Kubernetes Toleration objects). Capacity-style fields (`maxNodes`, `cpu`, `memory`, `gpu`) and the derived label-key convention are removed from this entity entirely. The `id` is the foreign-key-style reference from deployments and from the two default env vars; the `name` is presentation only and may be changed without breaking any deployment.
- **Deployment (surface refined)** — Holds a `nodePoolId` field that is either an id from a pool in current `NODE_POOLS` or null. On read, the response carries both `nodePoolId` (the persisted id, or null) and `nodePoolName` (the resolved display label from the current configuration; null when `nodePoolId` is null or when the id is dangling). Null is a normal, fully-supported value (rendered as "Any" in the FE) meaning "no pool primitives, default K8s scheduling" — it is always a legal selection regardless of `NODE_POOLS` configuration. The record stores only the id (or null); the pool's primitives, when applicable, are resolved by id at deploy time, so a pool's *internal* re-configuration takes effect on the next redeploy. There is no "effective vs stored" distinction for the id: what is stored IS what will be applied. The only way the stored `nodePoolId` changes after create is via an explicit update (FR-014). When the field is set by the create-time cascade (omitted payload, no default applicable), the cascade may resolve to null — that is the legitimate "no default, so Any" outcome, not a missing value.
- **Workload Classification** — A system-defined boolean attribute of every deployment type: a deployment is either a **model workload** (NIM, KServe-Inference) or not. Not stored on the deployment record — derived from the deployment's type at create time. Used only by FR-018's create-time cascade to decide whether `NODE_POOL_DEFAULT_MODEL` is eligible. Deploy-time logic never consults this classification.
- **Fallback Defaults** — Two platform-operator configuration values: `NODE_POOL_DEFAULT` (catch-all stamped onto any deployment whose create payload omits `nodePool`) and `NODE_POOL_DEFAULT_MODEL` (a narrower override that, when set, is stamped onto model workloads in place of the catch-all). Each is independently optional. The relationship is by-name: a default is a string pointer into `NODE_POOLS`, validated against the pool set at startup. **The defaults influence only the moment of stamping** — they are not consulted again after the deployment record is written. An admin change to either env var affects future creations only; existing records are immutable to env-var drift.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An operator can declare a node pool whose nodes are selected by a `matchExpressions` block (e.g. `accelerator-type In [nvidia-a100, nvidia-h100]`) without the deployment-manager needing a per-cluster convention — this expression appears verbatim on the deployed workload.
- **SC-002**: A pool that mixes `nodeSelector`, `affinity`, and `tolerations` results in a workload whose pod template carries all three sections — selector and affinity are AND-ed by the scheduler, tolerations are appended to whatever defaults the deployment type provides.
- **SC-003**: Any `NODE_POOLS` document that uses the removed fields (`maxNodes`, `cpu`, `memory`, `gpu`) causes startup to fail with an error message that names the offending field — no silent acceptance, no automatic upgrade path. The legacy `NODE_POOL_LABEL_KEY` environment variable is silently ignored if still present; operators who fail to remove it from their environment suffer no functional consequence because nothing reads it.
- **SC-004**: A deployment that previously scheduled via the derived `{labelKey: poolName}` affinity, when redeployed under this feature against a pool of the same name whose primitives are now explicit, lands on the same nodes (or a deliberately different set if the operator chose to change the targeting) without operator intervention beyond rewriting `NODE_POOLS`.
- **SC-005**: The `GET /api/v1/node-pools` response shape is reduced to declarative primitives only — an administrator inspecting the endpoint can read the pool definitions and predict exactly which scheduling slots will be set on a workload assigned to that pool, with no implicit conversions performed by the system.
- **SC-006**: An operator can administer a two-tier default with at most two environment variables: setting `NODE_POOL_DEFAULT` alone causes every new deployment whose create payload omits `nodePool` to be stamped with that pool; additionally setting `NODE_POOL_DEFAULT_MODEL` carves out NIM and KServe-Inference deployments to a separate (typically GPU-bearing) pool at create time.
- **SC-007**: A user inspecting any deployment via the API or the UI MUST see, in the `nodePool` field, the exact pool that the workload uses — there is no hidden default behaviour. Whether the value was typed by the user or stamped by the system at create time, the user can tell which pool they got and modify it explicitly if they want a different one.
- **SC-008**: An administrator who changes `NODE_POOL_DEFAULT` or `NODE_POOL_DEFAULT_MODEL` after restart MUST NOT observe any existing deployment's `nodePool` field changing as a result of the env-var change. Only deployments created after the change pick up the new default. Migrating an existing deployment to a new pool is always an explicit, per-deployment update.
- **SC-009**: "Any" (null `nodePool`) is a first-class user-selectable value regardless of operator pool configuration. Users can always opt out of pool-based scheduling — by selecting "Any" on create, by setting `nodePool` to null on update, or by submitting a create payload that omits `nodePool` when no fallback default applies. The FE picker always offers "Any" alongside configured pools. The operator does not need to declare a no-primitives pool to make this option available; conversely, the operator cannot disable it via this API.

## Assumptions

- **Verbatim projection, not merge.** The system replaces the workload's `nodeSelector` and `affinity` with the pool's values rather than merging them with any deployment-type defaults. `tolerations` are appended (deployment-type defaults + pool tolerations) because tolerations are independent of each other in Kubernetes semantics; replacing them would risk silently dropping operator-injected defaults such as those NIM / KServe operators add to their managed pods. If a deployment type already populates `nodeSelector` or `affinity` for its own reasons, those values are overwritten on pool projection — this matches Feature 016 behaviour and the user's intent that the pool's declaration is authoritative.
- **Full Kubernetes Affinity object accepted.** The user's example showed only `nodeAffinity`, but `affinity` accepts the full Kubernetes Affinity schema (`nodeAffinity`, `podAffinity`, `podAntiAffinity`) because (a) it costs nothing extra — the validation is delegated to the Kubernetes API model — and (b) some workloads benefit from pod-anti-affinity to spread across nodes inside a pool. Operators who only need `nodeAffinity` simply leave the other two unset.
- **Validation delegated to the Kubernetes API model.** Field-level validation of `nodeSelector`, `affinity`, and `tolerations` is performed by deserialising into the Kubernetes API types — the system does not re-implement schema checks. This means error messages on malformed config reflect the underlying deserialiser's vocabulary (e.g. "unknown field 'foo'"), which is acceptable because operators authoring these structures will recognise them.
- **YAML for configuration, JSON for the API response.** `NODE_POOLS` is consumed as YAML because operators already author K8s primitives (`affinity`, `tolerations`, `matchExpressions`) in YAML everywhere else — `kubectl`, Helm values, Deployment manifests — and the new nested shape is unreadable as a single-line JSON env-var value. YAML is a strict superset of JSON, so an operator who prefers a single-line JSON literal continues to work unchanged. Typed deserialisation into Kubernetes API model objects defuses YAML's notorious value-coercion footguns (the Norway problem, accidental booleans, octal-looking strings) because each field has a declared type. The `/api/v1/node-pools` HTTP response remains JSON — only the configuration input format is YAML.
- **No backward-compatible migration.** Because pool configuration changes already require an application restart and because the previous capacity fields (`maxNodes`, `cpu`, `memory`, `gpu`) are now sourced from live cluster state via Feature 020's `?includeUtilization=true`, there is no value in supporting both schemas concurrently. Stale config fails fast.
- **Deployment-type coverage unchanged.** Pool projection applies to the same deployment types Feature 016 already covered (MCP, Interceptor, Adapter, Application, NIM, KServe-Inference). For CRD-based types, projection lands on the CRD's pod-template equivalent. Adding new deployment types is out of scope for this feature.
- **`nodePool` field on Deployment is by-name, not by-value.** Deployments store only the pool name; `nodeSelector` / `affinity` / `tolerations` are not copied onto the deployment record. Pool primitives are resolved by name at deploy time from the current `NODE_POOLS` configuration. The operational consequence: when an admin edits a pool's primitives in `NODE_POOLS` (e.g. tightens an `affinity` block, adds a `toleration`) and restarts the application, the next redeploy of **every** existing deployment that references that pool by name picks up the new primitives — silently, with no per-deployment edit. This is the right behaviour for pool *contents* (a pool is conceptually a re-tunable scheduling preset), but it is asymmetric to the "stamped at create time" guarantee that governs pool *identity*: the deployment's chosen pool name does not migrate when defaults change, but the chosen pool's primitives do migrate when the pool itself is edited. Operators must be aware of this asymmetry — it is documented prominently in `docs/configuration.md` (see Dependencies).
- **Single pool per deployment.** Inherited from Feature 016 — multi-pool selection remains out of scope.
- **Cascade rather than two parallel slots.** Defaults are modelled as "catch-all plus model override" rather than "two independent category slots" so that the most common operator intent — "send everything sensible to one pool" — costs one env var instead of two. The trade-off: with only `NODE_POOL_DEFAULT` set, model workloads land on it too. Operators with model-serving workloads on dedicated hardware are expected to also set `NODE_POOL_DEFAULT_MODEL`. This is a deliberate ergonomics-over-safety choice; the misconfiguration is immediately visible (the first model deployment shows the catch-all pool in its `nodePool` field on create) and recoverable by setting the override.
- **Defaults stamped at create time, not resolved at deploy time.** When a deployment is created without an explicit `nodePool`, the system runs the cascade once, persists the result onto the deployment record, and returns it in the create response. From that point on the record is canonical — deploy-time logic, retrieval responses, and the FE all read the same stored value. The two consequences are deliberate and connected: (a) the user always sees on the FE the actual pool that will be used (no hidden defaults), and (b) an admin who later changes either env var does not silently migrate existing deployments. The alternative — resolve at deploy time — keeps the record sparse but makes deployments opaque ("which pool am I actually on?" requires reconciling stored-value vs env-var state) and creates an action-at-a-distance migration risk on env-var edits. For a tool whose users care about scheduling visibility, create-time stamping is the right call.
- **Distinguishing "field absent" from "field=null" in the payload.** Standard JSON semantics give a clean separator. On create, absent triggers the cascade; explicit null skips the cascade and is honoured verbatim (= "Any"). On update, absent means "leave unchanged"; explicit null means "set to null" (= "Any"). Both forms succeed regardless of pool configuration; only non-null names need to resolve in `NODE_POOLS`. Clients/SDKs that conflate absent and null must preserve the distinction in the wire format — collapsing them would make every routine save of an unrelated field accidentally re-trigger the cascade, which Story 4 explicitly forbids.
- **Pool list as the operator's catalogue, not whitelist.** `NODE_POOLS` defines a catalogue of named scheduling presets that users may select from. It is **not** an authoritative whitelist of acceptable scheduling outcomes — "Any" (null) is always a legitimate user selection regardless of operator configuration. The user experience is uniform (one picker showing configured pools plus "Any") and the operator's role is to provide good named presets and sensible defaults rather than to enforce mandatory pooling. This is a deliberate philosophical choice: pool selection is advisory in nature, not a security boundary (pods still run under default RBAC, network policies, etc.). Operators who need mandatory pooling enforce it through UI policy or organisational practice — not through this API.
- **Backend stays dumb about dangling references.** The backend's responsibilities for pool consistency are exactly two: (a) validate any non-null `nodePool` value against current `NODE_POOLS` at write time (FR-013, FR-014), and (b) fail redeploy honestly when a stored pool name no longer resolves. The backend does NOT compute a derived "pool exists" status on retrieval responses, does NOT offer an `?dangling=true` audit query, and does NOT implement a deprecation lifecycle on pool entries. Visualising dangling references is purely a FE concern — the FE has the deployment record (carrying its stored `nodePool`) and the listing response (carrying the current pool set) in hand, and joining them is a trivial client-side check. Putting that derivation in the backend would duplicate work the FE must do anyway (it still has to fetch the listing for the picker), without adding any capability the FE doesn't already have. The asymmetry is deliberate: backend stores facts and enforces invariants, FE composes views.
- **Model classification is binary and code-defined.** Only "model workload" is a named classification — there is no symmetric "non-model" category in the system's vocabulary, only "workloads to which `NODE_POOL_DEFAULT_MODEL` does not apply." Adding a third tier (e.g. batch workloads) in the future means another `NODE_POOL_DEFAULT_<TIER>` env var slotted into the create-time cascade — not a reshuffle of named categories.

## Dependencies

- **Feature 016 (Node Pool Selector)** — This feature supersedes the configuration shape and deploy-time scheduling logic introduced by 016. The deployment-side surface (the `nodePool` name field on Deployment, the validation that rejects unknown pool names, the redeploy semantics on pool change/clear) is retained.
- **Feature 020 (Node Pool Utilization)** — This feature reduces the static `/api/v1/node-pools` shape to declarative primitives only; the live capacity fields that 016 used to expose are now sourced from 020's `?includeUtilization=true` query parameter, which is unchanged.

## Documentation Deliverables

- **`docs/configuration.md` — Node Pool Configuration section MUST be rewritten as part of this feature**. The existing Feature 016 content (the `NODE_POOL_LABEL_KEY` env var, the JSON-encoded `NODE_POOLS` shape with `maxNodes` / `cpu` / `memory` / `gpu` per pool, the derived `{labelKey: poolName}` selector) is fully obsolete and MUST be replaced with documentation of the new shape: YAML-document `NODE_POOLS` carrying per-pool `nodeSelector` / `affinity` / `tolerations`, the two new fallback env vars (`NODE_POOL_DEFAULT`, `NODE_POOL_DEFAULT_MODEL`), and strict-mode parsing semantics. In addition to describing the shape, the rewrite MUST include an explicit operational note covering the by-name-at-deploy-time resolution: that a deployment stores only the pool name (not the primitives), and consequently that any admin edit to a pool's `nodeSelector` / `affinity` / `tolerations` in `NODE_POOLS` propagates on the next redeploy to every existing deployment referencing that pool. This is the intended behaviour (a pool is a re-tunable preset), but it is non-obvious — without the note, an admin tightening a pool's affinity may be surprised that existing deployments inherit the change automatically while default-env-var changes never migrate existing assignments. Both behaviours are correct and complementary, but the asymmetry must be made explicit in operator-facing documentation.
