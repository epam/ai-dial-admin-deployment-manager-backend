# BuildKit

## Purpose
This spec describes the BuildKit container image build engine — how the application uses BuildKit to build images from configured sources and push them to the registry.

Status: **Implemented**

## Key Terms
- **BuildKit**: A modern container image build toolkit (v0.27.1) used as the build engine, replacing the previous Kaniko-based approach.
- **Build context**: The source material for the build — either a Git Dockerfile source or a Docker image source.
- **Build pod**: The ephemeral Kubernetes pod in which BuildKit executes the build.
- **ImageWrapperBuildPipeline**: Pipeline that takes an existing Docker image, analyses its distro info, and builds a wrapper image around it. Used for MCP LOCAL transport Docker-source definitions and for Adapter/Interceptor/Application definitions.
- **SBOM**: Software Bill of Materials, generated by Syft for built images.

## Requirements

### Requirement: Build images from Git Dockerfile source
The system SHALL use BuildKit to build a container image by cloning a Git repository and executing a Dockerfile within it.

Status: **Implemented**

#### Scenario: Git Dockerfile build
- **WHEN** a build is triggered for an image definition with a `GitDockerfileImageSourceDto` (repository URL + Dockerfile path)
- **THEN** BuildKit clones the repository and builds the image using the specified Dockerfile

### Requirement: Build images from Docker image source
The system SHALL use BuildKit to create a container image based on a specified pre-existing Docker image reference.

Status: **Implemented**

#### Scenario: Docker image source build (wrapper)
- **WHEN** a build is triggered for an image definition with a `DockerImageSourceDto` requiring a wrapper build (MCP LOCAL transport)
- **THEN** BuildKit analyses the existing image and constructs a wrapper image from the specified base image via `ImageWrapperBuildPipeline`

#### Scenario: Docker image source copy
- **WHEN** a build is triggered for an image definition with a `DockerImageSourceDto` requiring only a copy (MCP REMOTE transport, Adapter, Interceptor, Application)
- **THEN** the image is copied into the managed registry via Skopeo through `ImageCopyPipeline` without invoking BuildKit

### Requirement: Built images pushed to the configured registry
After a successful build, the system SHALL push the resulting image to the registry configured by `DOCKER_REGISTRY`.

Status: **Implemented**

#### Scenario: Image pushed on success
- **WHEN** a BuildKit build completes successfully
- **THEN** the built image is tagged and pushed to the configured Docker registry

#### Scenario: Failed build does not push
- **WHEN** a BuildKit build fails
- **THEN** no image is pushed to the registry

### Requirement: Two build modes — root and rootless
The system SHALL support two BuildKit execution modes selectable per image definition via the `imageBuilder` field (`ImageBuilderDto`):

- **BUILDKIT** (root mode): uses `moby/buildkit:v0.27.1` image with `privileged: true` security context.
- **BUILDKIT_ROOTLESS** (rootless mode): uses `moby/buildkit:v0.27.1-rootless` image with `runAsUser: 1000`, `runAsGroup: 1000`, `appArmorProfile.type: Unconfined`, and `BUILDKITD_FLAGS: "--oci-worker-no-process-sandbox"`. No privileged mode required. This is the default when not explicitly specified.

Both modes use the `buildctl-daemonless.sh` command and produce a Docker tarball output (`type=docker`) that is subsequently pushed by a Skopeo sidecar container.

Status: **Implemented**

#### Scenario: Root mode build
- **WHEN** a build is triggered with `imageBuilder: BUILDKIT`
- **THEN** the build pod runs with the root BuildKit image and `privileged: true` security context

#### Scenario: Rootless mode build
- **WHEN** a build is triggered with `imageBuilder: BUILDKIT_ROOTLESS` (or no explicit imageBuilder)
- **THEN** the build pod runs with the rootless BuildKit image, non-root user (UID 1000), and no privileged flag

### Requirement: Build executes in an ephemeral Kubernetes pod
The system SHALL execute BuildKit builds inside ephemeral Kubernetes pods, tracked as disposable resources for cleanup.

Status: **Implemented**

#### Scenario: Build pod created
- **WHEN** a build is triggered
- **THEN** a BuildKit pod is created in Kubernetes and its reference is registered as a disposable resource

#### Scenario: Build pod cleaned up after completion
- **WHEN** a build reaches a terminal state (success or failure)
- **THEN** the build pod is eligible for cleanup and will be removed by the scheduled cleanup job

### Requirement: Build logs captured and available for streaming
The system SHALL capture all BuildKit output during the build and make it available for SSE streaming and later retrieval.

Status: **Implemented**

#### Scenario: Logs captured during build
- **WHEN** BuildKit produces log output during a build
- **THEN** all log lines are captured and stored against the image definition

#### Scenario: Stored logs available post-completion
- **WHEN** a build has completed and a client streams logs
- **THEN** the captured logs are replayed via the SSE log endpoint

## Implementation Notes
- Build pipelines (in `com.epam.aidial.deployment.manager.service.pipeline`):
  - `ImageBuildFromGitPipeline`    — clones Git repo, runs BuildKit base build, optionally runs wrapper build for LOCAL transport
  - `ImageWrapperBuildPipeline`    — analyses existing Docker image via `ImageAnalysisStep`, builds wrapper via `WrapperImageBuildStep`
  - `ImageCopyPipeline`            — copies existing Docker image to managed registry via `ImageCopyStep` (Skopeo)
- Build job specification: `com.epam.aidial.deployment.manager.service.pipeline.specification.ImageBuildAndPushJobSpecification`
- Image builder enum: `com.epam.aidial.deployment.manager.model.ImageBuilder` (BUILDKIT, BUILDKIT_ROOTLESS)
- BuildKit version: 0.27.1 (replaced Kaniko)
- SBOM generation: Syft
- Disposable resource tracking: see `kubernetes-cleanup` spec
- Related specs: `image-builds`, `container-registry`, `kubernetes-cleanup`
