## 1. Critical Changes (must apply before upgrade)

### 1.1 Deployment & Scaling — Node Pool Configuration

Node Pool Configuration is now generally available (the `[Preview]` tag has been removed).

> [!IMPORTANT]
> To use Node Pool Configuration for **Knative** deployments, the `pod-annotations`, `pod-affinity`, and `nodeselector` keys **MUST** be added to the Knative `config-features` ConfigMap. Without these keys, node pool scheduling hints are silently ignored for Knative workloads.

#### New Environment Variables

| Setting | Default | Required (Yes/No) | Description |
| :--- | :--- | :--- | :--- |
| `NODE_POOLS` | `""` | No | YAML list of node pool entries (id, name, description, `nodeSelector`, `tolerations`). |
| `NODE_POOL_DEFAULT` | `""` | No | Catch-all default pool ID stamped onto deployments created without an explicit `nodePoolId`. |
| `NODE_POOL_DEFAULT_MODEL` | `""` | No | Model-workload default pool ID for NIM and KServe-Inference deployments. Takes precedence over `NODE_POOL_DEFAULT`. |

**Note on node pool defaults:** Set `NODE_POOL_DEFAULT` only if you want every new deployment to land on a specific pool by default. Use `NODE_POOL_DEFAULT_MODEL` when GPU-backed model workloads (NIM, KServe-Inference) must be pinned to a separate pool from general workloads.

**Helm values.yaml:**
```yaml
manager:
  env:
    NODE_POOLS: '[{"id":"cpu-pool","name":"CPU pool","description":"Example CPU pool","nodeSelector":{"node-type":"cpu"}},{"id":"gpu-pool","name":"GPU pool","description":"Example GPU pool","nodeSelector":{"node-type":"gpu"},"tolerations":[{"key":"nvidia.com/gpu","operator":"Equal","value":"true","effect":"NoSchedule"}]}]'
    NODE_POOL_DEFAULT: "cpu-pool"
    NODE_POOL_DEFAULT_MODEL: "gpu-pool"
```

## 2. High-Priority Changes

### 2.1 Security & RBAC — API Key Authentication

DIAL Admin Deployment Manager can now accept an `Api-Key` header as an alternative credential alongside JWT/OIDC, validated against DIAL Core's `/v1/user/info` endpoint.

> [!NOTE]
> API key authentication is **disabled by default** (`API_KEY_ENABLED=false`). Existing JWT/OIDC-only deployments require no action.

#### New Environment Variables

| Setting | Default | Required (Yes/No) | Description |
| :--- | :--- | :--- | :--- |
| `API_KEY_ENABLED` | `false` | No | Accepts the `Api-Key` header as an alternative credential alongside JWT/OIDC. |
| `API_KEY_CORE_URL` | `""` | Yes (if `API_KEY_ENABLED=true`) | Base URL of DIAL Core used for validating API keys. |
| `API_KEY_ROLES_MAPPING` | `""` | Yes (if `API_KEY_ENABLED=true`) | JSON mapping of Core project-key role names to DM application roles. |
| `API_KEY_CACHE_TTL_SECONDS` | `60` | No | TTL for cached introspection results. |
| `API_KEY_CACHE_MAX_SIZE` | `10000` | No | Maximum entries in the introspection cache. |
| `API_KEY_REQUEST_TIMEOUT_MS` | `3000` | No | Per-call timeout for HTTP requests to Core's `/v1/user/info`. |
| `API_KEY_STARTUP_PROBE` | `true` | No | Probes Core's user-info endpoint at startup and aborts if unreachable. |

**Note on enabling API keys:** You must set this group of variables only if you want to allow service-to-service or project-key access via the `Api-Key` header. When `API_KEY_ENABLED=true`, both `API_KEY_CORE_URL` and `API_KEY_ROLES_MAPPING` are mandatory — without them the service cannot validate keys or assign roles.

> [!CAUTION]
> When `API_KEY_STARTUP_PROBE=true` (the default), the service will **fail to start** if DIAL Core's user-info endpoint is unreachable at boot. Ensure `API_KEY_CORE_URL` points to a reachable Core instance, or set `API_KEY_STARTUP_PROBE=false` to defer the check.

**Helm values.yaml:**
```yaml
manager:
  env:
    API_KEY_ENABLED: "true"
    API_KEY_CORE_URL: "http://dial-core"
    API_KEY_ROLES_MAPPING: '{"admin":"FULL_ADMIN","viewer":"READ_ONLY_ADMIN"}'
```

### 2.2 Security & RBAC — Build Job RBAC

It is now possible to **stop build jobs**, which requires an additional Kubernetes permission.

> [!IMPORTANT]
> The `deletecollection` verb for `jobs` **MUST** be added to all build roles. Without it, stopping a build job will fail with a permissions error.

**Example: Build Role with `deletecollection` permission**
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: dial-mcp-manager
  namespace: mcp-build
rules:
  - apiGroups:
      - "batch"
    resources:
      - jobs
    verbs:
      - get
      - list
      - watch
      - deletecollection
      - create
      - delete
  - apiGroups:
      - ""
    resources:
      - pods
      - pods/log
    verbs:
      - get
      - list
      - watch
  - apiGroups:
      - ""
    resources:
      - secrets
      - configmaps
    verbs:
      - get
      - create
      - delete
```

## 4. Deprecations, Migrations, and Removals

### 4.1 Removed properties

> [!IMPORTANT]
> `config.rest.security.default.allowedRoles` and `providers.*.allowed-roles` have been **removed** in this release. In `0.16.0` these were deprecated and auto-mapped to `FULL_ADMIN`; that backward-compatibility shim is gone. If you still rely on them, you **must** migrate to `roles-mapping` before upgrading or admin access will break.

#### Removed properties

| Variable | Description |
| :--- | :--- |
| `config.rest.security.default.allowedRoles` | Removed. Use the `roles-mapping` configuration (`SECURITY_ROLES_MAPPING` / `providers.*.roles-mapping`) to map IdP roles to DM application roles (`FULL_ADMIN`, `READ_ONLY_ADMIN`). |
| `providers.*.allowed-roles` | Removed. Use the per-provider `providers.*.roles-mapping` override instead, which merges with the default mapping (provider takes precedence). |
